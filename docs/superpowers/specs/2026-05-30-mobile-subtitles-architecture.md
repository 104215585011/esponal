## 一、 背景与核心 Reframe

在当前的设计中，网页视频播放页（`/watch`）高度依赖 **Chrome 扩展程序** 进行字幕的抓取和上传（Ingestion）。这导致了两个致命的局限性：
1. **移动端体验断层**：移动端浏览器（Safari, Chrome 移动版）无法运行 Chrome 扩展程序，导致移动端用户打开视频无法获得字幕。
2. **冷启动门槛高**：新用户必须先在桌面端安装扩展程序、抓取并上传，别人才能消费该视频。

### 核心 Reframe：策划频道的批量预抓

YouTube 对反爬的封锁主要卡在**“大规模、实时抓取任意视频”**。而 Esponal 的定位是**“精选策划内容”**。
我们有且仅有几个策划好的核心 YouTube 频道（如 *Dreaming Spanish*, *Easy Spanish* 等），视频总量是**有限、已知且可枚举的**。

**结论**：我们不需要在用户请求视频时去“实时抓取”，而是通过**后台慢速批处理脚本（批量预抓）**把这些策划频道的历史视频字幕拉取下来，并存入 Redis/数据库。所有用户（包括移动端）在打开这些精选视频时，直接从我们的服务端获取字幕。

## 二、 字幕获取三层防御架构

为了让用户获得“零门槛、高质量”的字幕体验，我们把架构拆成**两条独立通道**：**离线填充**（把字幕预先灌进缓存）与**在线请求路径**（用户读字幕）。

> ⚠️ **关键原则**：yt-dlp 是**离线批处理**，只负责把字幕**提前填进 Redis**；它**绝不在用户请求路径里实时运行**。否则就退回到「实时抓任意视频 → 触发 YouTube 反爬」的老坑，本架构的 reframe 就白做了。

```
【离线填充通道 — 批处理，提前跑】
  批处理 worker(yt-dlp,定时/慢速) ──抓策划频道所有视频的人工 es 字幕──> 写入 Redis/DB

【在线请求路径 — 用户实时只读】
        用户请求 /api/subtitle
                 │
        1. 查缓存(Redis/DB) ├── 命中 ──> 返回(策划视频 99% 走这里)
                 │
               未命中(off-catalog 零散视频)
                 ▼
        2. 托管 API 兜底(Apify/Supadata) ├── 成功 ──> 返回 + 回填缓存
                 │
                失败
                 ▼
        3. 返回 no_subtitle(并由运营用扩展事后补抓)
```

### 1. 批量预抓层 (Primary - 核心质量控制，离线)
- **实现手段**：自建 worker 集成 `yt-dlp`，**离线、定时、低频慢速**(必要时加代理)批量抓取策划频道下所有视频的人工西语字幕。**不在请求路径触发**。
- **命令行参数**：`yt-dlp --write-subs --sub-langs es,es-419 --sub-format json3 --skip-download <url>`
- **存储**：json3 解析为现有标准 Cue **`{start, dur, text}`(秒)**(同 `SubtitleCue` / 扩展 `parseJson3.js`)，写入 Redis/DB，与现有 ingest 格式完全对齐。

### 2. 实时托管 API 兜底层 (Secondary - 零散视频支持)
- **实现手段**：当用户访问不在精选频道内的零散视频（Off-catalog videos）且缓存未命中时，服务端在后台发起一次实时抓取，请求 Supadata 或 Apify 托管字幕 API。
- **优点**：零运维，专门的代理基建对抗 YouTube 反爬，按次收费。

### 3. 运营扩展上传层 (Tertiary - 运营工具)
- **重新定位**：Chrome 扩展从“每个用户的必备客户端工具”退役为“**内部运营/编辑专用工具**”。
- **用途**：当自动预抓和托管 API 均失效，或者某些视频的 AI 字幕质量较差需要人工修正时，管理员可以用桌面端扩展直接抓取当前浏览的字幕并上传覆盖。

## 三、 实施规划

> 移动上线分**两条并行轨**：A. 字幕服务化(让字幕不依赖扩展)，B. 原生壳打包(让产品上应用商店)。A 是 B 的前置价值——壳做好了字幕喂不上也没意义，所以 A 的 POC 最先做。

### 轨 A：字幕服务化

**A1. POC 验证（1.5 天）— ticket `SUBS-001`**
- 验证自建 `yt-dlp` 从数据中心 IP 抓策划频道人工 es 字幕的**质量**(对比 Redis 已有扩展版 ground truth)、**反爬**(是否被封/需代理)、**成本**(对比托管 API)。
- 这是整个移动上线的**可行性闸门**，先做。

**A2. 批处理 Worker + 管道集成（3-5 天）— ticket `SUBS-002`（POC 通过后）**
- 定时同步脚本：拉策划频道所有视频 → 解析 → 写 Redis(envelope 标 `source: "harvest"`)。
- `/api/subtitle` 请求路径维持「缓存 → 托管 API 兜底」，**不接 yt-dlp**。
- worker 跑哪初期不挑：定时 GitHub Action 或本机 cron 起步，稳了再容器化(Fly/Railway)。

**A3. 移动端字幕体验 + 扩展退役（2-3 天）**
- 移动端去掉扩展安装引导，直接读服务端字幕；之前说的「轮询自动优先扩展/harvest 字幕」也在此落地。
- Chrome 扩展权限/文案改为**运营专属**(`EXT_INGEST_TOKEN` 共享密钥问题随之消失——客户不装扩展)。

### 轨 B：原生壳上架（Capacitor）

> 用户已定交付形态=上应用商店。因 Esponal 是**重服务端 Next.js**(API 路由/NextAuth/force-dynamic)→ **无法静态导出** → Capacitor 走 **WebView 加载远程 Vercel URL** 模式，后端继续留 Vercel。

**B1. 移动布局适配**（轨 A3 共用）：视频/逐字稿竖叠或 tab、触屏交互、manifest/viewport。
**B2. Capacitor 套壳**：iOS/Android 壳加载生产 URL。
**B3. 加原生价值 + 过审**：推送通知、离线课、深链、首屏图标——避免被 App Store 4.2「纯网页壳」拒；提交上架。

> 轨 B 细节可在 POC 通过、字幕服务化路线确定后单开一份 `2026-XX-native-shell-packaging.md` spec。
