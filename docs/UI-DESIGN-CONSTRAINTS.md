# Esponal UI 设计禁区清单

> 本文件是**所有 UI ticket 的强制约束**。任何 UI 实现违反以下条目即视为不通过，无论视觉多好看。
> 创建：2026-05-27 · 由 Claude1 (PM) 维护

---

## 为什么有这个清单

Esponal 的目标用户是「想学西语但被传统语言 App 折磨过的人」。市面上的 Duolingo / Babbel 都用游戏化机制制造焦虑——连续打卡、XP 条、每日提醒、成就徽章。我们不做。

这份清单把"不做什么"明确写下来，避免每张 ticket 重复争论。

---

## 禁区清单（违反即不通过）

### 1. 不要游戏化数字

❌ 连续打卡天数（streak counter）
❌ XP 条 / 经验值
❌ 等级显示（Lv. 12）
❌ 每日学习进度环 / 进度百分比放在显眼位置
❌ "今天还差 X 分钟"、"还差 X 词达标"
❌ 任何让用户产生「不学就输了」感觉的元素

✅ 可以做：「已收藏 89 词」这类**静态、客观、低优先级**的展示
✅ 可以做：用户主动进入"复习"页面后，内部的进度可视化

### 2. 不要伪 AI 标签

❌ 标注「AI 智能分析」「AI 词形拆解」「AI 学习建议」等，**实际上是查表/规则**的功能
❌ 用 ✨ Sparkles 图标暗示某段是 AI 生成
❌ 给 `tryConjugateVerb` 这种规则匹配冠以「AI」之名

✅ 可以做：真正用 LLM 的功能（如 `/talk` 对话）可以标注
✅ 可以做：用平实的名字称呼功能：「词形分析」「变位查询」「释义查询」

### 3. 不要"已掌握"反向视觉

❌ 给已掌握词加删除线（视觉上等于"被划掉"）
❌ 把已掌握词变得很难看（褪色到几乎看不见）
❌ 给已掌握词加 ✓ 完成标记（暗示"这个词死了"）

✅ 可以做：已掌握词保持普通文本样式，但**不再加任何高亮**（已掌握 = 不需要标注）
✅ 可以做：在词库筛选页面给出明确的状态标签

### 4. 不要 SRS / 间隔重复术语

❌ 「已入 SRS 排程」
❌ 「Anki 卡片」、「Leitner box」
❌ 「记忆曲线」、「遗忘曲线」
❌ 算法时间表（"3 天后复习"、"明天再来"）

✅ 可以做：内部用 SRS 算法没问题，但**用户可见的文案**用人话：「该复习了」「最近遇到 3 次」

### 5. 不要不必要的提醒和压力

❌ 弹窗"你今天还没学习！"
❌ 顶部红点 / 角标"未完成的任务"
❌ 邮件 / 推送"长时间未登录"
❌ 倒计时压力（"距离掌握还差 X 天"）

✅ 可以做：用户主动进入应用，应用乐于服务；用户离开，应用沉默

### 6. 不要游戏化奖励

❌ 徽章、奖杯、勋章墙
❌ 「连续 7 天奖励」
❌ 完成任务的庆祝动画（confetti、纸屑）
❌ 排行榜、好友对比

✅ 可以做：完成阅读后的安静"已读"状态
✅ 可以做：用户自己看到的"已收藏 N 词"这种客观数字

### 7. 不要中文母语者不友好的设计

❌ 只用英文+西文标注，中文缺位
❌ 阴阳性 / 变位概念没有中文类比解释
❌ 错误提示用英文 / 西文

✅ 可以做：所有 UI 文案中文优先，西语作为学习目标语言出现

## 如何使用本清单

### Gemini1（UI 总监）
- 每张 UI ticket 开工前先扫一遍本清单
- 设计稿 / 实现里出现的元素逐条对照
- 不确定的找 Claude1 确认

### Claude1（PM）
- 写 ticket 时在「约束」段落引用本文件
- 验收时把每条禁区作为否决项
- 发现新的违反案例，增补本清单

### Codex2（QA）
- 端到端测试时如发现违反清单的元素，记入 report
- 但**不阻塞功能性测试**——清单违反由 Claude1 / Gemini1 处理

## 历史背景：触发本清单的具体案例

2026-05-27 Web Gemini 生成的查词面板原型出现以下违反：

1. 已收藏按钮文案「已入 SRS 排程」→ 违反 §4
2. 智能分析模块标题「AI 语法拆解」→ 违反 §2（实际是查表）
3. 已掌握词加删除线 → 违反 §3
4. 阅读页底部"A1 84% / A2 11%"百分比徽章 → 违反 §1

这四条都不是配色问题，是框架问题。无论换成什么颜色都还是违反。
