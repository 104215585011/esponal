# WEB-019 — YouTube 配额优化：watch 相关视频改用 channel 接口（消灭 search.list 误用）

**优先级**：P1（配额成本，正在快速消耗）
**区域**：`src/app/watch/page.tsx`（相关视频获取）+ 可能 `src/lib/youtube.ts` / channel 逻辑
**实现**：Codex1
**测试**：Codex2
**无 UI**：是 → Claude1 → Codex1 → Codex2
**预估**：0.5 天

---

## 背景

YouTube Data API 配额按方法计费,**search.list = 100 单位/次**,而 channel 路径(channels.list + playlistItems.list + videos.list)≈ **3 单位**。

当前 watch 页"相关视频"逻辑（`src/app/watch/page.tsx` ~line 80-100）:
- **精选频道**(curatedChannels,3 个):走 `fetchChannelVideos(curated.id)` channel 接口(~3u)✅
- **非精选频道**:回落 `/api/youtube/search?q=channelTitle&maxResults=8`(**search.list 100u**)❌

"相关视频"的语义是 **"这个频道的其他视频"(more from this channel)**——这本该用频道上传列表(channel uploads),跟"搜索"无关。非精选频道却烧 100u,既贵又不准（search 按关键词,不等于同频道）。

> 今日配额快速消耗(3,433/10,000)的**主因**是缓存被手动清空 + 缓存 key bump 到 v2 → 冷启动全量重灌。本票解决**结构性浪费**(search 误用),并加固"别清缓存"。

---

## 要做的改动

1. **非精选频道相关视频改走 channel 接口**：
   - 先解析当前视频的 `channelId`（用 `videos.list` part=snippet 拿 `snippet.channelId`,1u；若现有取时长/embeddable 的 videos.list 调用可顺带 part 加 snippet,则零额外成本）。
   - 用该 `channelId` 走现有 channel 上传列表逻辑(`fetchChannelVideos` / `/api/youtube/channel`,~3u),取"同频道其他视频"。
   - **仅当拿不到 channelId 时**才回落 search.list（极少数）。
   - 结果:相关视频 100u → ~4u,且更准(真·同频道)。
2. **保留 search.list 仅用于 `/search` 用户主动搜索**（那是发现型功能,需新鲜,24h 短缓存不变）。
3. **加固"别清缓存"**：在 youtube 缓存相关代码/文档加注释说明——**不要在常规运维清 `youtube:*` 缓存、不要随意 bump 缓存 key 版本**,每清一次都会触发全量 search.list 重灌烧配额;若必须失效,错峰/分批。

- [ ] 缓存代码处有"勿清空/勿 bump key"的注释提醒。
