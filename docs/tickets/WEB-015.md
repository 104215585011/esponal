# WEB-015 — /watch 页与 SiteHeader 宽度断层修复

**优先级**：P2（视觉一致性，不影响功能）
**区域**：web / layout
**依赖**：WEB-009 ✅（设计令牌）、WEB-014 ✅（BackLink）

---

## 背景

PM 在 1920px 显示器上发现：

- **SiteHeader 用 `max-w-screen-xl`（1280px）居中限宽** —— 左右两侧各有 ~320px 留白
- **`/watch` 主内容**（视频 + 字幕 2 列布局）**无 max-width**，铺满全屏 ~1910px
- 结果：**header 比下面内容窄**，视觉上"被截了一刀"

PM 实测：按 F12 打开 DevTools 后视口缩到 ~1430px，断层不明显 → 用户体验和默认全屏时差异巨大。

---

## 项目现状盘点

```
max-w-screen-xl（1280px）：SiteHeader / /learn / /lectura / /extension / homepage / /learn/[slug]
max-w-5xl（1024px）：       /grammar / /grammar/[slug]
max-w-3xl（768px）：         /lectura/[slug] / /learn/phase-1
无 max-width：               /watch（左右双列布局，刻意撑满？）
```

主路径全部用 `max-w-screen-xl` 是事实上的项目约定，**只有 /watch 没遵守**。

---

## 修复方向（3 选 1）

### 方案 A：/watch 加 `max-w-screen-xl`（最低变更）

只改 `src/app/watch/page.tsx` 一行：

```tsx
<div className="relative flex flex-col lg:h-[calc(100vh-58px)] lg:flex-row ...">
// 改为：
<div className="relative mx-auto w-full max-w-screen-xl flex flex-col lg:h-[calc(100vh-58px)] lg:flex-row ...">
```

- **优**：1 行改动；视觉与全站对齐；最低风险
- **劣**：1920px 屏上视频列从 ~1200px 缩到 ~800px，**视频变小**——对学习场景可能不友好

### 方案 B：去掉 SiteHeader 的 `max-w-screen-xl`（全站铺满）

改 `src/app/components/web/SiteHeader.tsx`，header 不再限宽。

- **优**：1 行改动；wide-screen 视频列保持现状最大
- **劣**：1920px 屏上 logo 离左边、avatar 离右边各 ~900px，header 显得**空荡稀疏**；可能破坏所有页面的视觉重心

### 方案 C：全站升级到 `max-w-screen-2xl`（1536px，推荐）

改 SiteHeader + 所有用 `max-w-screen-xl` 的页面 → `max-w-screen-2xl`。

- **优**：wide-screen 利用率好（1920px 屏上左右各 192px 留白，舒适）；保持视觉一致；视频列 ~970px，比方案 A 的 800 大；text 内容（lectura/learn）也能透气
- **劣**：触动 6+ 文件；要做小范围回归（其他页面不要破坏阅读宽度）

---

## PM 倾向：方案 C

理由：
- 当下用户都是 16:9 大屏，1280 是过去 13" MacBook 时代的舒适宽度，2026 年偏小
- 视频学习场景，越大的播放器越能看清口型/嘴型，价值确凿
- text 类页面（lectura/learn 课文）影响小——`max-w-2xl`（672px）和 `max-w-3xl`（768px）这些 inner-content 的 max-width 不变，外壳变宽不影响文本可读性

但方案 C 涉及多个页面，是否引入视觉副作用（比如 home/extension 落地页本来在 1280 是精心设计的留白）需要 Claude2 评审。

---

## 评审请求

Claude2 重点判断：

1. **方案 C 是否会破坏**已经精心打磨过的页面（home / extension / learn / lectura 列表）？这些页面的卡片网格、hero 留白可能在 1280px 是设计好的
2. **是否有第 4 种方案**——比如 `/watch` 独立用 `max-w-[1600px]`（既不和 header 对齐也不全屏），text-heavy 页面继续用 1280？
3. 如果方案 C：是否需要同时把 `/grammar` 的 `max-w-5xl` 也对齐？还是它有意保留更窄阅读宽度？
4. 1536px 是不是合适的目标，还是应该 `max-w-[1600px]` 或 `max-w-screen-3xl`（需查 tailwind 是否原生支持）？

---

## 不在本票范围

- 重新设计页面网格（只调容器宽度）
- 响应式 breakpoint 调整
- 移动端布局（移动端无 lg: 限制不受影响）
- `/lectura/[slug]` 阅读页保持 `max-w-3xl`（这是有意的窄阅读区，不动）

---

## 最终方案：方案 C 修订版（Claude2 评审通过 2026-05-21）

### Claude2 评审核心修订

1. **新增设计令牌**：在 `tailwind.config.ts` 的 `extend.maxWidth` 加 `'app-shell': '96rem'`（1536px，对应 `screen-2xl`），所有外壳容器统一用 `max-w-app-shell` 语义化别名，防未来令牌漂移
2. **`/watch` 特殊处理**：不在外层 `<main lg:h-screen lg:overflow-hidden>` 加 max-width（会破坏 100vh 沉浸感）；在**内层** `<div className="...lg:flex-row...">`（line 100）加 `mx-auto w-full max-w-app-shell`，让左右两列在 shell 内分配宽度
3. **不动** `/grammar`、`/lectura/[slug]`、`/learn/phase-1` —— 这些是阅读型窄宽（max-w-5xl / max-w-3xl），1024 / 768px 是 typography 最佳行长，有意保留
4. **回归核查**：1920px 截图 home / watch / extension / learn / lectura 五页，确认 header 与下方内容左右对齐、无横向滚动条、卡片网格无塌陷
