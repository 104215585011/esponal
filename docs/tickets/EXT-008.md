# EXT-008 — Chrome 扩展字幕采集（EXT-006 复活精简版）

**优先级**：P1（直接决定 `/watch` 字幕质量天花板）
**区域**：extension / web
**依赖**：WEB-003 ✅（/watch 已存在）、WEB-004 ✅（subtitle 路由已存在）、OPS-002 ✅（rate limit 基础设施）

---

## 背景

我们 `/watch` 页字幕来源目前是 Apify YouTube 抓取器 + 本地 Whisper 兜底。Apify 受 YouTube 反爬（PO Token 验证）影响，对大量教学类视频要么拿不到字幕、要么拿到降级版本。Spike 验证（2026-05-20，HAR 文件分析 + youtubei.js PO Token 测试）确认：

- **西语助手（Eudic Chrome 扩展）**之所以字幕质量好，**就是**因为它在用户已登录的浏览器里跑 content script，直接 fetch `youtube.com/api/timedtext`，命中带 PO Token 的官方 JSON3 接口
- **服务器端纯 PO Token 方案**（路径 2B）经 spike 验证不可行——多数视频返回 `LOGIN_REQUIRED: Sign in to confirm you're not a bot`
- 用户**主动浏览 YouTube** + 我们扩展被动采集，是**唯一**能稳定拿到等同西语助手质量字幕的路径

我们曾在 EXT-006/EXT-007 实现过这套方案，2026-05-18 commit `81c8ad3` 整体删除，原因：
1. Chrome 138+ 阻断 `--load-extension` 让 Playwright 自动批量预采集不可行（**这条限制依然存在，不复活该路径**）
2. 当时给"想象中的测试者"用，认为安装门槛过高（**现在场景明确：用户就是自己，门槛不再是阻力**）

本票复活原 EXT-006 的"用户安装扩展 + 浏览 YouTube 时自动采集"路径，**不**复活 EXT-007 的 Playwright 批量预采集。

---

## 用户故事

1. 用户去 `chrome://extensions` 加载 Esponal 扩展（开发者模式手动加载，或未来从 Chrome Web Store 装）
2. 用户照常在 YouTube 看西语视频，扩展**全程透明**采集字幕回 Esponal DB
3. 用户切回 Esponal `/watch?v=同 ID` → 立即命中高质量字幕（YouTube 官方轨道，与西语助手等价）
4. 用户打开**没访问过 YouTube 的视频**时 → Esponal 静默走 Apify/Whisper 兜底；兜底也失败时，UI 提示用户"在 YouTube 看一遍即可自动归档"

---

## 三个决策（已与 PM 确认）

| 决策 | 选项 |
|------|------|
| Ingest 接口认证 | **保留 `X-Esponal-Ingest-Token`**，token 在扩展打包时注入；视作垃圾流量过滤而非安全密码；主要防护靠 IP rate limit |
| `/watch` 字幕命中失败时 | **三段式**：先查 Redis 库存（扩展灌入的）→ 没有就走 Apify → Apify 也空就显示"在 YouTube 看一遍"提示 |
| 扩展触发时机 | **任意** `youtube.com/watch*` 自动触发；只采集 `es-*` 轨道；无西语轨道时静默放弃 |

## 技术方案

### 1. 扩展侧：`extension/harvest.js`（content script）

**Bridge 模式**（与原 EXT-006 一致，已验证可行）：

```js
// 1. 注入一段 <script> 到页面，运行在页面 JS 上下文里
//    读取 window.ytInitialPlayerResponse.captions.playerCaptionsTracklistRenderer.captionTracks
// 2. 用 window.postMessage 把数据扔回 content script（隔离世界）
// 3. content script 收到后，对每个 es-* 轨道 fetch 它的 baseUrl + "&fmt=json3"
//    fetch 时 credentials: "include" → 浏览器自动带 YouTube cookie + PO Token
// 4. 解析 JSON3 → cue 数组 → POST 到 /api/subtitle/ingest
```

**采集触发**：每次落到 `https://www.youtube.com/watch*` 页面时跑一次。用 `window.__esponalHarvested = videoId` 防同一视频重复采集。

**最小阈值**：cue 数 < 5 不上报（极短或异常轨道丢弃）。

### 2. 扩展侧：`extension/parseJson3.js`

把 YouTube JSON3 `events[]` 结构（包含 `tStartMs`、`dDurationMs`、`segs[].utf8`、`tOffsetMs`）解析成我们标准 `{ start, dur, text }[]`。直接复用旧 EXT-006 实现（commit `2a281ce`）。

### 3. 扩展侧：`extension/manifest.json`

新增一条 content script 注册：

```json
{
  "matches": ["https://www.youtube.com/watch*"],
  "js": ["dist/harvest.js"],
  "run_at": "document_idle"
}
```

`host_permissions` 已含 `https://www.youtube.com/*`，无需变更。

### 4. 扩展侧：`extension/scripts/build.mjs`

esbuild 把 `harvest.js` + `parseJson3.js` 打成单个 `dist/harvest.js`。打包时通过 `--define` 注入：

```
ESPONAL_APP_ORIGIN  → 默认 https://esponal.example.com（占位，实际生产 URL）
EXT_INGEST_TOKEN    → 从环境变量 EXT_INGEST_TOKEN 注入
```

本地开发时 `ESPONAL_APP_ORIGIN=http://localhost:3000 EXT_INGEST_TOKEN=dev-token npm run build` 即可。

### 5. 扩展侧：`extension/popup.html` + `popup.js`

popup 只新增**一块**："上次采集" 信息，从 `chrome.storage.local.get("lastSubtitleHarvest")` 读：

```
最近采集
─────────────────
视频 ID: lZXrGDXsD-4
语言: es
条数: 58
时间: 2 分钟前
```

无数据时显示 "尚未采集任何字幕，去 YouTube 看几个西语视频试试"。其余 popup 内容不动。

### 6. 服务器侧：`src/app/api/subtitle/ingest/route.ts`

直接复刻旧 EXT-006 版本（commit `2a281ce`），关键参数：

- `X-Esponal-Ingest-Token` header 校验
- IP 级 rate limit（用 `ingestLimiter`）
- payload 大小上限 500KB
- cue 数量 5 ~ 5000
- 单 cue text 长度 < 500、dur 0~600 秒
- `videoId` 字符集 `[\w-]+` 长度 ≤ 32
- `lang` 格式 `^[a-z]{2}(?:-[A-Za-z0-9]+)?$`
- 写 Redis key `subtitle:v4:${videoId}:${lang}:auto`，TTL 30 天
- 已有该 key 时**不覆盖**（防止低质量采集覆盖已有高质量）
- 返回 `{ success, cueCount, written: boolean }`

### 7. 服务器侧：`src/lib/ratelimit.ts`

复活 `ingestLimiter`：每 IP 60 秒 30 次（用户一次浏览只触发一次 ingest，30 次已经很宽松）。

### 8. 服务器侧：`/watch` 三段式 fallback（**新逻辑，不在旧 EXT-006 范围**）

修改 `src/app/api/subtitle/route.ts` 的字幕获取流程：

```
1. 先查 Redis subtitle:v4:${videoId}:${lang}:auto
   ↳ 命中 → 返回（这一步现状已有）
2. 未命中 → 调 Apify（现状已有）
3. Apify 返回 0 cue → 在响应 body 加 hint 字段：
   { cues: [], hint: { reason: "no_subtitle", suggestion: "extension" } }
4. 客户端 /watch 页面读 hint，在 transcript 区域显示提示卡：
   "这个视频还没有归档高质量字幕。在 YouTube 上看一遍即可，
    扩展会自动采集到 Esponal。" + [在 YouTube 打开] 按钮
```

`/watch` 客户端组件 `TranscriptPanel.tsx` 增加空字幕状态的 hint UI（约 20 行 JSX）。

### 9. 扩展打包与分发

- `npm run package`（in `extension/`）打 zip → `public/extension/esponal-extension.zip`
- 用户从 `/extension` 落地页下载 zip → Chrome `chrome://extensions` 开发者模式 → "加载已解压的扩展程序"
- 现有 `/extension` 落地页（EXT-005 ✅）的安装步骤说明保持，无需新页面

### 10. 环境变量

`.env.example` 新增：
```
EXT_INGEST_TOKEN=""
ESPONAL_APP_ORIGIN="http://localhost:3000"
```

生产 Vercel env 配置同名变量（PM 部署前手动加）。

## 不在本票范围

- **EXT-007 Playwright 批量预采集**——Chrome 138+ 限制依然存在，不复活
- 上传扩展到 Chrome Web Store 发布（用户手动加载就行）
- 多语言扩展支持（只处理 es-* 轨道）
- 服务器端 PO Token 生成（spike 已证明不可行）
- 字幕版本管理（多次采集合并 / 用户编辑订正等）
- 把 ingest 数据归到具体用户账户（当前是匿名采集到全局缓存，不绑用户）
- 浏览器扩展的更新提示机制
- 离线模式下的扩展行为（无网络时静默放弃）
