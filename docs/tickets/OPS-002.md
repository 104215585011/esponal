# OPS-002 — API 限流（防配额烧穿）

**优先级**：P0 | **负责人**：Codex1 | **日期**：2026-05-16
**前置依赖**：无

---

## 背景

当前后端三个对外接口完全无限流：
- `/api/translate`（每次调用 = 1 次 Tencent TMT 计费）
- `/api/vocab/lookup`（首次调用 = 1 次 DashScope GLM-5 推理 ~510 tokens）
- `/api/vocab/add`（每次写 Prisma DB）

匿名用户、爬虫、或者前端 bug（无限重试循环）都能在几分钟内烧掉一天的免费配额。Tencent / DashScope 一旦透支会自动停服，影响所有真实用户。

本 ticket 加 Upstash Rate Limit（项目已用 Upstash Redis），按 IP + userId 双维度限流。

---

## 一、限流策略

| 接口 | 维度 | 阈值 | 理由 |
|---|---|---|---|
| `/api/translate` | IP + userId | 60 req / 分钟 | 一段视频 transcript 翻译初始 ~30 条 cue，留 2x buffer |
| `/api/vocab/lookup` | IP + userId | 30 req / 分钟 | 用户最快点词手速 ~1/秒，留 2x buffer |
| `/api/vocab/add` | userId 必有 | 20 req / 分钟 | 加生词不可能比这更频繁 |
| `/api/youtube/search` | IP | 20 req / 分钟 | 防爬 |
| `/api/youtube/channel` | IP | 30 req / 分钟 | 首页频道流自然访问 |

**IP 维度**：从 `request.headers.get("x-forwarded-for")` 或 `x-real-ip` 提取，多个 IP 取第一个。Vercel 自动注入。

**userId 维度**：从 `getServerSession()` 拿，未登录就只按 IP。

**双维度算法**：IP 和 userId 任一超限就拒绝。这样既防匿名爬虫，又防一个用户用不同 IP 攻击。

## 二、技术方案

### 依赖

```bash
npm install @upstash/ratelimit
```

`@upstash/redis` 项目已经有。

### 核心模块

新增 `src/lib/ratelimit.ts`：

```typescript
import { Ratelimit } from "@upstash/ratelimit";
import { Redis } from "@upstash/redis";

const redis = Redis.fromEnv(); // 复用现有 KV_REST_API_URL / KV_REST_API_TOKEN

export const translateLimiter = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(60, "1 m"),
  prefix: "rl:translate",
  analytics: false,
});

export const lookupLimiter = new Ratelimit({ ... });
export const addLimiter = new Ratelimit({ ... });
export const searchLimiter = new Ratelimit({ ... });
export const channelLimiter = new Ratelimit({ ... });

export function getClientIp(request: Request): string {
  const xff = request.headers.get("x-forwarded-for");
  if (xff) return xff.split(",")[0].trim();
  return request.headers.get("x-real-ip") ?? "unknown";
}

export async function checkRateLimit(
  limiter: Ratelimit,
  request: Request,
  userId: string | null
): Promise<{ allowed: boolean; reset: number }> {
  const ip = getClientIp(request);
  const ipResult = await limiter.limit(`ip:${ip}`);
  if (!ipResult.success) return { allowed: false, reset: ipResult.reset };

  if (userId) {
    const userResult = await limiter.limit(`user:${userId}`);
    if (!userResult.success) return { allowed: false, reset: userResult.reset };
  }

  return { allowed: true, reset: 0 };
}
```

### 接入每个路由

模板：

```typescript
import { checkRateLimit, translateLimiter } from "@/lib/ratelimit";
import { getServerSession } from "next-auth";
import { getAuthOptions } from "@/lib/auth";

export async function POST(request: Request) {
  const session = await getServerSession(getAuthOptions()).catch(() => null);
  const userId = session?.user?.id ?? null;

  const rl = await checkRateLimit(translateLimiter, request, userId);
  if (!rl.allowed) {
    return NextResponse.json(
      { error: "rate limited", retryAfterSec: Math.ceil((rl.reset - Date.now()) / 1000) },
      {
        status: 429,
        headers: { "Retry-After": String(Math.ceil((rl.reset - Date.now()) / 1000)) }
      }
    );
  }

  // ...existing logic
}
```

### 前端处理

`TranscriptPanel` 的 `translateCue` 已经有 retry 逻辑，遇到 429 应：
- 读取 `Retry-After` header
- 退避 `Retry-After + 抖动` 后重试
- 不要把 429 当成 degraded 立刻显示原文

`LookupCard` 遇到 429：
- 显示"查询过于频繁，请稍后再试"，不弹错误
- 不消耗当前 form 的 cache

## 三、降级安全

如果 Upstash 限流本身挂了（网络/服务故障），`checkRateLimit` 应**放行**而不是拦截——宁可短暂无限流也不能整站不能用。在 `try/catch` 里把错误吞掉，返回 `{ allowed: true }`。
